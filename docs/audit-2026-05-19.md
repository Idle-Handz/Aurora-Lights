# Codebase audit — 2026-05-19

Combed through actively maintained code (Aurora.App services and pages, Aurora.Components, CSS) for bugs and QoL issues. Decompiled legacy code in Aurora.Logic was not the focus.

This file lists the issues that were NOT auto-fixed because each requires either a design choice, a behavioral decision, or a non-trivial refactor. See "Already fixed" at the bottom for the obvious ones that were applied directly.

## High priority

### 1. XML saves are in-place, no atomic rename — partial writes corrupt character files
**Where:** [Aurora.App/Services/CharacterFileSaveExtensions.cs](Aurora.App/Services/CharacterFileSaveExtensions.cs) — `SaveCurrency`, `SaveTextEdits`, `SaveInventoryItems`, `SaveSession`, `SaveSourceRestrictions` all use `new XmlTextWriter(path, …)` which truncates the destination then writes.

**Risk:** If the process is killed, the disk fills, the device loses power, or Android revokes external-storage access mid-write, the `.dnd5e` file is left half-written and unreadable. Players lose the whole character, not just the session edit. The Session page calls `SaveSession`/`SaveCurrency` on every HP pip click — high write frequency.

**Fix sketch:** write to `path + ".tmp"`, then `File.Move(tmp, path, overwrite: true)` on success. On Windows that's atomic; on Android it's safe at the FAT level. Wrap the writer in `try/catch` so the temp file is deleted on failure.

**Why I didn't auto-apply:** five call sites need to change together, and Android external storage (content://) may require platform-specific handling — wants a small dedicated review.

### 2. `SilentLoadSession` swallows all exceptions and returns default state
**Where:** [Aurora.App/Services/CharacterFileSaveExtensions.cs:413](Aurora.App/Services/CharacterFileSaveExtensions.cs#L413) — `catch { return new SessionState(); }`

**Risk:** Combined with issue #1, a corrupted XML silently resets HP, death saves, spell slots, and custom resources to defaults on the next load. The player sees a "full HP" character with no warning.

**Fix sketch:** log via `DebugLogService.Instance.LogException(ex, "LoadSession")` so the issue surfaces in the Console, and consider surfacing a one-time `Snackbar` via the caller (Characters page or wherever `LoadSession` is invoked).

**Why I didn't auto-apply:** decision needed about user-visible messaging.

### 3. Portrait files use the picker's original filename — collisions silently overwrite
**Where:** [Aurora.App/Components/Pages/CharacterDetail.razor:782-787](Aurora.App/Components/Pages/CharacterDetail.razor#L782)

```csharp
string fileName = Path.GetFileName(pick.FileName);
string destPath = Path.Combine(portraitsDir, fileName);
// ... overwrites destPath unconditionally
```

**Risk:** Two characters whose users both pick `portrait.png` (very common — phone screenshots, default downloads) silently share a single file. Changing one character's portrait will change the other.

**Fix sketch:** prefix the filename with the character identifier (or use a guid + retain extension):
```csharp
string ext = Path.GetExtension(pick.FileName);
string fileName = $"{tab.File.DisplayName}_{Guid.NewGuid():N}{ext}";
```

**Why I didn't auto-apply:** there's already a parked design (`project_portrait_feature` memory) for a richer portrait picker — this fits with that work and shouldn't be a one-off patch.

## Medium priority

### 4. `CheckForUpdatesAsync` collapses "no update" and "error" into a single non-success result
**Where:** [Aurora.App/Services/ContentService.cs:124-146](Aurora.App/Services/ContentService.cs#L124) returns `(bool Updated, string Message)`. Settings UI at [Aurora.App/Components/Pages/Settings.razor:1031-1041](Aurora.App/Components/Pages/Settings.razor#L1031) always sets `Severity.Info` regardless of outcome, because the success/error distinction is lost.

**Risk:** Network exceptions display as a friendly "Info" message; the user can't tell whether content actually updated, was already current, or failed.

**Fix sketch:** return an enum (`Updated | UpToDate | Error`) or `(bool Success, bool Updated, string Message)` and switch severity in the page based on the outcome.

### 5. `CharacterTabService` is not thread-safe; mutated from background callbacks
**Where:** [Aurora.App/Services/CharacterTabService.cs](Aurora.App/Services/CharacterTabService.cs)

`_tabs` is a `List<CharacterTab>`. `OpenTab`/`CloseTab`/`MarkDirty` are called from MAUI UI and Blazor handlers, but `TabsChanged` event handlers in pages dispatch back into the list via `InvokeAsync(...)`. There's no explicit lock or thread affinity. So far it works because Blazor Server keeps a single sync context, but a future feature that mutates from `Task.Run` (e.g. background autosave) would race.

**Fix sketch:** document the "UI thread only" contract on the class, OR lock `_tabs` and the active-tab field for all mutations.

### 6. `LongRest` fallback uses `_s.CurrentHp` as max HP
**Where:** [Aurora.App/Components/Pages/Session.razor:526](Aurora.App/Components/Pages/Session.razor#L526)

```csharp
_s.LongRest(_snap?.MaxHp ?? _s.CurrentHp);
```

If `_snap` is null at the moment of the click (shouldn't happen given the page-level guard, but defensive), the rest "restores" HP to whatever current HP is rather than full. Defensive code that's actively wrong.

**Fix sketch:** the page already early-returns when `_snap is null`, so this fallback is dead — just pass `_snap.MaxHp` and let it NRE if the invariant is broken (or `return` if you want to keep the guard).

### 7. Settings → ReloadContentAsync navigates away from the Settings page
**Where:** [Aurora.App/Components/Pages/Settings.razor:1062](Aurora.App/Components/Pages/Settings.razor#L1062) — `Nav.NavigateTo("/")` after reload success.

**Risk:** The user is in the middle of configuring content packs; reloading and landing on the Characters page is disorienting. They may want to verify the reload picked up new sources by toggling more.

**Fix sketch:** stay on `/settings`. The tab list is empty already (closed by reload), so no per-character pages are affected.

### 8. Build page ability method always defaults to Manual on tab activation
**Where:** [Aurora.App/Components/Pages/Build.razor:595](Aurora.App/Components/Pages/Build.razor#L595) — `_abilityMethod` field initializer.

The standard-array `_arrayAssignments` initializer at lines 666-672 does detect the valid array and prefills, but the method selector stays on Manual, hiding that detection from the UI.

### 9. AutoSave on note textareas fires per keystroke (`@oninput`)
**Where:** [Aurora.Components/Shared/CharacterEquipmentWorkspace.razor:223,232,241](Aurora.Components/Shared/CharacterEquipmentWorkspace.razor#L223)

Each keystroke in equipment/treasure/quest notes triggers `OnNoteChanged` → `MarkDirty` → `TabsChanged` → re-render of MainLayout, NavMenu, and every page subscribed to TabsChanged. Multiple Razor components rebuild per character per keystroke. The save itself is in-memory only (no disk write per keystroke), but the event fan-out is significant.

**Fix sketch:** switch the textareas to `@onchange` (fires on blur) so the dirty mark and re-render happen once per edit session. The trade-off: dirty indicator lags until blur, which is mild.

### 10. `ChangeAmountAsync` in Equipment page mutates without explicit `StateHasChanged`
**Where:** [Aurora.App/Components/Pages/Equipment.razor:303-317](Aurora.App/Components/Pages/Equipment.razor#L303)

Other handlers (OpenGearSlotDialog, UnequipGear, OpenAddItemDialog, RemoveItemAsync, EditItemAsync, OpenStartingEquipmentAsync) all end with `StateHasChanged()`. `ChangeAmountAsync`, `ToggleEquippedAsync`, `ToggleAttunedAsync`, `ChangeCoinAsync`, `ChangeNoteAsync` do not.

In practice, the EventCallback wrapper re-renders the parent automatically, so this is usually fine. But it's inconsistent — if anyone refactors to call these from elsewhere (e.g. a keyboard shortcut), they'd skip the render.

**Fix sketch:** either add `StateHasChanged()` consistently, or document the EventCallback assumption at the top of the page.

## Low priority / nitpicks

### 11. `_subStepCts` cycling task can outlive disposal briefly
**Where:** [Aurora.App/Components/Pages/Build.razor:541-552](Aurora.App/Components/Pages/Build.razor#L541)

The background `Task.Run` checks `_disposed` only between iterations. If `Dispose` fires during a delay, the task wakes, observes the cancel, and exits cleanly — but there's a small window where `InvokeAsync` is queued after disposal. The InvokeAsync handler also checks `_disposed`, so this is safe; just noting the layered guard.

### 12. `parts[0][0]` in CharacterBrowser.GetInitials assumes non-empty parts
**Where:** [Aurora.Components/Shared/CharacterBrowser.razor:244-250](Aurora.Components/Shared/CharacterBrowser.razor#L244)

`string.Split(' ', RemoveEmptyEntries | TrimEntries)` does remove empty entries after trim, so `parts[0]` should always be non-empty. The top-level `IsNullOrWhiteSpace` check covers the "all whitespace" case. Safe today, but a brittle invariant to leave undocumented.

`entry.Level.ToString().ToUpper()[..3]` works for "Info"→"INF", "Warning"→"WAR", "Error"→"ERR". If a future `LogLevel` value has a name under 3 characters, this throws ArgumentOutOfRangeException. Minor — current enum is closed.
